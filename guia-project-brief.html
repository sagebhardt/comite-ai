Cómo Escribir un Buen Project Brief

Guía práctica para que tu postulación al Comité AI sea clara, completa, y se apruebe rápido

Un buen project brief no necesita ser largo — necesita ser claro. El comité evalúa decenas de propuestas, y las que se aprueban más rápido son las que responden las preguntas correctas sin rodeos. Esta guía te muestra exactamente qué incluir y cómo escribirlo.

Regla de oro: Si alguien que no conoce tu proyecto lee tu brief y entiende qué quieres hacer, por qué, y cuánto cuesta — está bien escrito.

Los 7 Componentes de un Buen Brief

1. El Problema Obligatorio

Qué es: La situación actual que quieres resolver. No la solución — el problema.

Pregunta clave: ¿Qué dolor existe hoy y quién lo sufre?

Mal "Queremos usar IA para mejorar el proceso de soporte."
Bien "El equipo de soporte recibe ~120 tickets/día. El 60% son preguntas repetitivas (estado de pedido, políticas de devolución). Cada agente gasta 3-4 horas/día en estas consultas, dejando poco tiempo para casos complejos. El tiempo promedio de respuesta es 4.2 horas."
Tip: Incluye números siempre que puedas. "Muchos tickets" no dice nada. "120 tickets/día, 60% repetitivos" dice todo.
2. El Objetivo de Negocio Obligatorio

Qué es: El resultado que esperas lograr, expresado en términos de negocio (no técnicos).

Pregunta clave: ¿Qué cambia concretamente si este proyecto funciona?

Mal "Implementar un chatbot con Claude para soporte."
Bien "Reducir el tiempo de respuesta a consultas repetitivas de 4.2h a menos de 5 minutos, liberando al equipo para resolver los tickets complejos que hoy se demoran 24h+."
Error común: Confundir el objetivo con la solución. "Implementar un chatbot" es una solución. "Reducir el tiempo de respuesta en un 80%" es un objetivo. Describe el qué, no el cómo.
3. La Solución Propuesta Obligatorio

Qué es: Cómo piensas resolver el problema usando IA. Aquí sí es técnico.

Pregunta clave: ¿Cómo funciona tu solución y qué tecnología usa?

Mal "Vamos a usar Claude para responder preguntas automáticamente."
Bien "Chatbot interno que usa Claude API (Sonnet) con RAG sobre nuestra base de conocimientos de soporte (~200 artículos en Zendesk). El usuario escribe su pregunta en un widget web, Claude busca artículos relevantes y genera una respuesta. Si la confianza es baja, escala a un agente humano. Backend en Python/FastAPI, desplegado en Cloud Run."

Incluye en tu descripción:

• Qué modelo de IA usarás (Claude, Gemini, etc.) y por qué
• De dónde saca los datos la IA
• Cómo interactúa el usuario con la solución
• Qué pasa cuando la IA no sabe la respuesta (fallback)
• Stack técnico (lenguaje, framework, infraestructura)
4. Datos y Seguridad Obligatorio

Qué es: Qué datos toca tu proyecto y qué tan sensibles son.

Pregunta clave: ¿Qué información procesa la IA y quién puede verla?

Mal "Usa datos de la empresa."
Bien "Accede a: (1) artículos de Zendesk (internos, no sensibles), (2) historial de tickets (contiene nombres y emails de clientes = PII). Los datos se procesan via API de Claude — no se almacenan respuestas. Acceso restringido al equipo de soporte (8 personas) via SSO."

Clasifica tus datos:

NivelEjemplosImplicancia
PúblicoInfo del sitio web, catálogos, precios publicadosSin restricciones adicionales
InternoProcesos, manuales, artículos de soporteNo compartir fuera de la empresa
ConfidencialVentas, márgenes, estrategia, datos financierosRequiere controles de acceso
RestringidoPII de clientes, datos regulados, contratosRequiere aprobación legal + controles estrictos
5. Métricas de Éxito Obligatorio

Qué es: Cómo sabrás si el proyecto funcionó. Números concretos con los que te comprometes.

Pregunta clave: ¿Qué medirás y cuánto tiene que mejorar para considerarlo un éxito?

Mal "Mejorar la satisfacción del cliente."
Bien "KPIs a 90 días: (1) Tiempo de respuesta a tickets repetitivos: de 4.2h a <5 min, (2) % de tickets resueltos sin humano: target 40%, (3) CSAT en respuestas del bot: >4.0/5.0, (4) Costo por ticket: de $2.800 CLP a $400 CLP."
Estructura recomendada: Para cada métrica, incluye el valor actual (baseline), el valor objetivo (target), y en cuánto tiempo esperas lograrlo.
6. Herramientas y Recursos Obligatorio

Qué es: Qué herramientas, servicios y recursos necesitas para tu proyecto. El comité se encargará de evaluar los costos asociados.

Pregunta clave: ¿Qué herramientas, APIs o servicios necesitas, y cuántas horas-persona requiere?

Mal "Vamos a usar IA."
Bien "Herramientas: Claude API (modelo Sonnet, ~50K tokens/día), hosting en Cloud Run (1 instancia), Cloud Storage para documentos, Vertex AI Embeddings para búsqueda semántica, monitoreo con Cloud Monitoring. Horas-persona: 40h setup inicial (dev senior), 4h/mes mantención."
Sé específico en las herramientas. Detalla qué servicios planeas usar y con qué volumen estimado. El comité utilizará esta información para evaluar los costos y clasificar tu proyecto.
7. Riesgos y Mitigación Obligatorio

Qué es: Qué podría salir mal y qué harás al respecto. El comité valora la honestidad sobre la venta.

Pregunta clave: ¿Cuáles son los 3 principales riesgos y cómo los manejarás?

Mal "Riesgo bajo, no vemos problemas."
Bien "Riesgos: (1) Alucinaciones — el bot podría dar info incorrecta sobre políticas. Mitigación: respuestas solo basadas en artículos verificados + disclaimer + botón de escalamiento. (2) Costos mayores a lo estimado si el volumen sube. Mitigación: alerta automática al 80% del presupuesto + rate limiting. (3) Baja adopción por el equipo. Mitigación: piloto de 2 semanas con 3 agentes antes del rollout completo."

Todo proyecto tiene riesgos. Decir "no hay riesgos" no genera confianza — genera preocupación de que no pensaste bien el proyecto.

Bonus: Lo que Diferencia un Brief Excelente

Estos elementos no son obligatorios, pero los mejores briefs los incluyen y se aprueban más rápido.

Elementos que Suman Puntos
Timeline con hitos: No solo "Marzo a Mayo" sino "Semana 1-2: prototipo, Semana 3-4: piloto con 3 usuarios, Semana 5-8: iteración, Semana 9: rollout". Demuestra que tienes un plan real.
Alternativas consideradas: "Evaluamos también usar Gemini Flash (más barato pero menos preciso en español) y una solución de Zendesk AI (más cara, $2M CLP/mes, menos personalizable)." Muestra que no es capricho.
Plan B / criterios de abandono: "Si a los 30 días el CSAT está bajo 3.5/5.0, pausamos y reevaluamos antes de invertir más." Le da confianza al comité de que no se van a quemar recursos indefinidamente.
Verificación de duplicación: "Revisé el Project Tracker — el proyecto de Marketing (#AI-012) usa Claude para generar contenido, pero no tiene componente de soporte. Hablé con María (dueña) y confirmamos que no hay overlap."
Impacto cuantificado: "Si reducimos el costo por ticket de $2.800 a $400 CLP en 70 tickets/día (los repetitivos), el ahorro es ~$5M CLP/mes. ROI positivo desde el mes 2."

Ejemplo Completo de un Buen Brief

Así se ve un brief que el comité puede evaluar y aprobar rápido:

Ejemplo: Chatbot de Soporte Tier 1

Problema
El equipo de soporte (8 personas) recibe ~120 tickets/día. El 60% son preguntas repetitivas sobre estado de pedidos, políticas de devolución y horarios. Cada agente dedica 3-4 horas/día a estas consultas. El tiempo promedio de primera respuesta es 4.2 horas, y nuestro SLA target es 2 horas. Estamos incumpliendo el SLA en un 35% de los tickets.
Objetivo
Resolver automáticamente el 40% de los tickets repetitivos, reduciendo el tiempo de respuesta a menos de 5 minutos para esas consultas, y liberando ~20 horas-persona/día para tickets complejos.
Solución
Chatbot web integrado en el portal de soporte. Usa Claude API (Sonnet) con RAG sobre los 200 artículos de Zendesk. El usuario describe su problema, Claude busca artículos relevantes y genera una respuesta personalizada. Si la confianza es baja o el usuario no queda satisfecho, escala a un agente humano con el contexto de la conversación. Stack: Python 3.11, FastAPI, Cloud Run, Vertex AI Embeddings para el índice de búsqueda.
Datos
Fuentes: artículos de Zendesk (internos, no sensibles) + historial de tickets (contiene nombres y emails de clientes = PII). Clasificación: Restringido por PII. Los datos se envían a Claude API pero no se almacenan respuestas. Acceso limitado a equipo de soporte via SSO corporativo.
Métricas de Éxito (a 90 días)
Tiempo de respuesta tickets repetitivos: 4.2h → <5 min. Tickets resueltos sin humano: 0% → 40%. CSAT en respuestas del bot: target >4.0/5.0. Costo por ticket repetitivo: $2.800 → $400 CLP. Cumplimiento de SLA general: 65% → 85%.
Herramientas y Recursos
Claude API Sonnet (~50K tokens/día). Cloud Run (1 instancia). Cloud Storage para base de conocimiento. Vertex AI Embeddings para búsqueda semántica. Cloud Monitoring para alertas. Setup: 40 horas dev senior (2 semanas). Mantención: 4 horas/mes.
Riesgos
(1) Alucinaciones: respuestas incorrectas sobre políticas. Mitigación: respuestas solo de artículos verificados + disclaimer + botón "Hablar con agente". (2) Sobrecosto si sube el volumen. Mitigación: alerta al 80% del presupuesto + rate limiting en 200 consultas/día. (3) Baja adopción. Mitigación: piloto 2 semanas con 3 agentes, iteración antes de rollout. Criterio de abandono: si CSAT <3.5 a los 30 días, pausamos.

Checklist Pre-Envío

Antes de enviar tu postulación, verifica que puedas responder "sí" a cada punto:

Antes de hacer clic en "Enviar"

El problema está descrito con datos concretos (no solo "mejorar X")
El objetivo es medible y tiene un número asociado
La solución técnica está descrita con suficiente detalle para que alguien la entienda sin preguntarte
Clasifiqué los datos correctamente (público / interno / confidencial / restringido)
Si uso PII, explico qué controles tendré
Mis métricas tienen baseline actual + target + plazo
Listé todas las herramientas y servicios que necesito, con volúmenes estimados
Listé al menos 2-3 riesgos reales con mitigaciones concretas
Revisé el Project Tracker y no hay duplicación
Alguien que no conoce mi proyecto podría entender este brief

Errores Frecuentes que Atrasan la Aprobación

ErrorPor qué atrasaSolución
Brief demasiado vago El comité tiene que pedir más info, lo que atrasa 1+ semana Usa esta guía. Si tienes dudas, más detalle es mejor que menos
Confundir objetivo con solución No se entiende el valor de negocio El objetivo responde "qué logramos". La solución responde "cómo lo hacemos"
Herramientas sin detalle El comité no puede evaluar costos ni viabilidad sin saber qué usarás Lista cada herramienta, API o servicio con el volumen estimado de uso
"No hay riesgos" Genera desconfianza — todo proyecto tiene riesgos Sé honesto. Mejor 3 riesgos con mitigación que 0 riesgos
No verificar duplicación Pueden existir esfuerzos paralelos que no conoces Revisa el Tracker. Si hay algo similar, referencia y diferencia
Métricas sin baseline "Mejorar 50%" no significa nada si no sé el punto de partida Siempre: "De X actual a Y objetivo en Z tiempo"

Requisitos de Documentación MVP

Si tu proyecto implica construir un MVP, tu postulación debe incluir (o comprometerse a entregar) la siguiente documentación técnica para facilitar la migración a producción por el equipo TI.

Documentación obligatoria del MVP

Base de datos: Tablas, campos, tipos de dato, claves primarias/foráneas, relaciones e índices
Servicios externos: Nombre, tipo de integración (REST/Webhook/SDK), datos enviados y recibidos, método de autenticación, justificación
Diagrama de flujo: Diagrama simple del flujo del sistema (entrada → proceso → resultado)
Endpoints: Listado de endpoints si el MVP expone APIs
Variables de entorno: Lista de todas las variables de entorno necesarias (sin incluir valores reales)
Recordatorio técnico:
  • El MVP debe usar Python 3 + SQLite — no usar NoSQL
  • La estructura de BD debe ser compatible con migración a MySQL/MariaDB
  • No duplicar funcionalidades del proyecto base (autenticación, correos, reportes, etc.)
  • No escribir directamente en SAP S/4 HANA ni en el Data Warehouse
  • Ningún servicio externo pasa a producción sin evaluación del comité
